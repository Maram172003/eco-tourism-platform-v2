1. Architecture actuelle — état des lieux

La plateforme dispose déjà d'une architecture en 3 couches construites lors de l'onboarding :

👤 Provider
user_id (PK) full_name, photo region, country bio, website activity_types[ ] sustainability_score
→
🏢 Organization
id (PK) provider_id (FK) name, logo, bio, history region, address, zone, country lat, lng phone, email, website instagram, facebook, tiktok photos[ ], videos[ ] certifications[ ] eco_labels[ ]
→
⚡ ProviderActivity
id (PK) provider_id (FK) organization_id (FK) level: primary | secondary category (eco_tour, hebergement…) subtypes[ ] years_experience MongoDB (ActivityDetails) fields { } — champs spécifiques photos { subtype: url[ ] } certifications[ ]
→
📋 Offer (existant)
id (PK) author_id → provider.user_id title, description, price offer_type, offer_subtype confirmation_mode details { } (JSONB) images[ ], region capacity, cancellation_policy sustainability_score status: pending|approved|rejected ⚠ organization_id manquant
⚠️ Problème identifié : L'entité Offer est liée au Provider via author_id, mais pas à l'Organization. Or c'est l'organisation qui publie des offres (elle a le nom, l'adresse, les photos, les certifications). De plus, les offres ne sont pas connectées aux ProviderActivities déclarées lors de l'onboarding.
✅ Ce qui existe déjà dans l'entité Offer : offer_type, offer_subtype, details (JSONB), confirmation_mode, capacity, min_group_size, max_group_size, deposit_percentage, cancellation_policy, sustainability_score. Le modèle de données est déjà très complet — seul organization_id manque.

2. Ce qui change : Offre liée à l'Organisation

2.1 Principe général

Une offre est publiée par une organisation, pas directement par le prestataire. L'organisation est l'entité publique visible (avec son nom, logo, adresse). Le prestataire reste le propriétaire en coulisse.

🔗 Nouveau lien

Offer → organization_id (FK vers Organization)
L'offre hérite automatiquement :
• Région et adresse de l'org
• Lat/lng du point de rencontre
• Nom et logo de l'org pour l'affichage public

⚡ Lien optionnel activité

Offer → provider_activity_id (FK optionnel vers ProviderActivity)
Permet de pré-remplir le formulaire depuis une activité existante (type, sous-type, certifications, expérience…)

2.2 Les 3 chemins pour créer une offre

A

Depuis une activité existante (recommandé)

Le prestataire clique "Créer une offre" sur une carte d'activité dans son profil (onglet Activités ou À propos). Le formulaire se pré-remplit avec le type, sous-type, région, et certaines données spécifiques déjà saisies lors de l'onboarding.

B

Nouvelle offre libre

Via le bouton "+ Publier une offre" classique. Le prestataire choisit le type, sous-type, et remplit tout le formulaire depuis zéro. Mais la région et le point de rencontre se pré-remplissent depuis l'organisation.

C

Depuis un type non couvert par les activités déclarées

Par exemple, publier une offre "Volontariat" alors que l'org n'a pas déclaré de ProviderActivity de type volontariat. Formulaire libre complet — cohérent avec la section 3 du document original.

4. Modèle de données — modifications minimales

4.1 Entité Offer — 2 colonnes à ajouter

-- Colonnes EXISTANTES dans offer.entity.ts (déjà implémentées) :
id, author_id, title, description, price, duration
offer_type, offer_subtype, fulfillment_mode, confirmation_mode, price_type
capacity, booking_deadline_hours, confirmation_deadline_hours
deposit_percentage, details (jsonb), images[], inclusions
region, meeting_point, meeting_lat, meeting_lng
min_group_size, max_group_size, min_age
cancellation_policy, sustainability_score, status

-- 2 colonnes À AJOUTER :
organization_id : uuid → FK vers organizations.id (nullable)
activity_id : uuid → FK vers provider_activities.id (nullable, optionnel)
✅ Grâce à synchronize: true en TypeORM, ces 2 colonnes seront créées automatiquement au redémarrage du backend — aucune migration manuelle nécessaire.

4.2 Entité Reservation — état actuel et ajouts

L'entité Reservation doit stocker les champs spécifiques à chaque type de réservation (dates d'arrivée/départ pour hébergement, nombre de participants pour activité…).

-- Colonnes à prévoir dans Reservation :

offer_id : uuid → FK Offer

5. Formulaire de création d'offre adapté (4 étapes)

Le formulaire actuel dans le profil prestataire est un formulaire simple en 1 écran. La proposition ci-dessous le transforme en 4 étapes progressives, enrichies du pré-remplissage depuis les activités déclarées.
ÉTAPE 0

🚀 Point de départ

  • Afficher les activités de l'org (orgActivities) comme cartes cliquables
  • "Créer depuis cette activité" → pré-remplit les étapes suivantes
  • Ou "Nouvelle offre libre" → formulaire vide
  • Badge de niveau : Principale / Secondaire
ÉTAPE 1

📂 Type & Sous-type

  • Sélection du type d'offre (11 types)
  • Sélection du sous-type selon le type
  • Si depuis activité existante → pré-sélectionné automatiquement
  • Indicateur visuel "Depuis votre activité XXX"
ÉTAPE 2

📝 Informations générales

  • Titre de l'offre *
  • Description *
  • Prix (TND) + type de prix *
  • Durée (texte libre)
  • Région ← pré-remplie depuis org
  • Point de rencontre ← pré-rempli depuis org
  • Carte GPS ← lat/lng depuis org
  • Photos (upload, min 1)
  • Inclusions (texte)
  • Politique d'annulation
ÉTAPE 3

⚙️ Détails & Confirmation

  • Champs spécifiques au type/sous-type ← partiellement pré-remplis
  • Capacité, min/max participants, âge min
  • Mode de confirmation : Instantané / Sur demande / Avec acompte
  • Si acompte : % de dépôt
  • Délai de réponse (si sur demande)
  • Questionnaire durabilité

7. Processus de confirmation et de réservation

7.1 Modes de confirmation — choisi par le prestataire à l'étape 3

ModeValeur backendDescriptionAdapté pour
✅ Confirmation instantanée instant Réservation confirmée automatiquement dès soumission Éco-tour, Activité, Artisanat, Transport
⏳ Sur demande manual Prestataire accepte/refuse dans le délai défini (24h/48h) Hébergement, Restaurant, Bien-être, Volontariat
💰 Avec acompte conditional X% payé en ligne pour bloquer, solde sur place Hébergement longue durée, Éco-tour multi-jours

7.2 Flux complet de la réservation (côté voyageur)

1

Découverte de l'offre

Depuis la page Destinations ou le profil d'un prestataire → clic sur "Réserver cette offre"

2

Solo ou Groupe

Solo → 1 personne | Groupe → inviter des amis par recherche. Le prix total est divisé automatiquement entre les participants du groupe.

3

Formulaire de réservation spécifique au type

Dates, participants, préférences alimentaires, contact d'urgence, notes… (voir section 8 ci-dessous)

4

Récapitulatif & confirmation

Résumé complet : offre, organisation, date, participants, prix total, politique d'annulation. Le voyageur confirme.

5

Traitement selon le mode de confirmation

• Instantané → statut "Confirmée" immédiatement
• Sur demande → statut "En attente" → prestataire notifié → accepte/refuse → voyageur notifié
• Avec acompte → paiement partiel → statut "Confirmée"

6

Notifications

• Email de confirmation au voyageur (avec QR code)
• Notification au prestataire (en attente ou confirmée)
• Rappel 24h avant l'expérience
• Invitation email aux amis du groupe

7

Après l'expérience

Statut → "Terminée" → questionnaire post-expérience envoyé → note et avis publiés sur le profil de l'organisation

7.3 Statuts de la réservation

StatutValeurSignificationAction possible
