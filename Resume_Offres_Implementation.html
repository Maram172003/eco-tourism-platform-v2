Règle clé : Une offre est publiée par une Organisation et peut être liée à une ProviderActivity existante (primaire ou secondaire). Depuis une activité avec N sous-types, on peut créer N offres distinctes (une par sous-type).

✅ Ce qui existe déjà dans l'entité Offer : offer_type, offer_subtype, details (JSONB), confirmation_mode, capacity, min/max_group_size, deposit_percentage, cancellation_policy, sustainability_score, meeting_lat/lng, images, status.

2. Relation Activité → Offres

Une ProviderActivity avec plusieurs sous-types peut générer autant d'offres que de sous-types déclarés. Chaque offre couvre un ou plusieurs sous-types selon le besoin :

ProviderActivity "Hébergement Montagne Verte"
subtypes: [dortoir, suite, chambre_double]
│
├── Offre 1 → "Nuit en dortoir" (1 sous-type : dortoir)
├── Offre 2 → "Suite panoramique" (1 sous-type : suite)
└── Offre 3 → "Pack Famille" (2 sous-types : suite + dortoir)

2.1 Offre avec plusieurs sous-types — 2 modes

🔀 Mode Variantes (le voyageur choisit)

Le voyageur sélectionne un sous-type parmi les options proposées. Chaque variante peut avoir son propre prix.


Exemple : "Hébergement au choix"
○ Dortoir — 45 TND/nuit
● Suite — 120 TND/nuit

📦 Mode Package (tout inclus)

L'offre comprend tous les sous-types sélectionnés simultanément. Un seul prix global.


Exemple : "Pack Famille Forêt"
☑ Suite parents + ☑ Dortoir enfants
Prix total : 165 TND/nuit

3. Formulaire de création d'offre — 5 étapes

0

Point de départ — Choisir une activité existante

Grille de toutes les orgActivities (primaires et secondaires). Badge "X offre(s) déjà créée(s)" sur chaque carte. Ou bouton "Nouvelle offre libre" sans lien à une activité.

1

Sous-type(s) de cette offre

Sélection d'un ou plusieurs sous-types déclarés dans l'activité. Choix du mode : Variantes (voyageur choisit) ou Package (tout inclus). Si sous-type déjà couvert par une offre existante → badge d'avertissement.

2

Informations commerciales (uniquement ce qui n'existe pas dans l'onboarding)

Titre de l'offre · Prix (par nuit / personne / groupe) · Durée · Inclusions (ce que comprend le prix) · Politique d'annulation. Tout le reste (photos, région, adresse, lat/lng) est pré-rempli depuis l'organisation.

3

Détails spécifiques — 3 couches par sous-type

Pour chaque sous-type sélectionné : données onboarding en référence (lecture seule) + champs supplémentaires propres à l'offre + validation des contraintes en temps réel. Voir section 4.

4

Calendrier de disponibilité + Mode de confirmation

4 modes de disponibilité + 3 modes de confirmation. Voir sections 5 et 6.

4. Les 3 couches de données à l'étape 3

🔒 Couche 1 — Données onboarding (référence, lecture seule)

Affichées comme contexte pour le prestataire. Servent de contraintes de validation — le prestataire ne peut pas créer une offre qui dépasse ce qu'il a déclaré.

Ex: "Capacité déclarée : 60 couverts" / "Niveaux acceptés : Débutant, Intermédiaire" / "Langues guides : FR, AR"

✏️ Couche 2 — Détails supplémentaires de l'offre (à remplir)

Champs plus détaillés et spécifiques à cette offre précise — au-delà de ce que l'onboarding capture. Varie selon le sous-type.

Stockés dans Offer.details (JSONB)

💰 Couche 3 — Informations commerciales (étape 2)

Titre, prix, durée, inclusions, politique d'annulation — ce qui est propre à chaque offre commerciale.

5. Système de contraintes — validation depuis l'onboarding

Quand le prestataire remplit l'offre, les données de l'activité liée définissent des limites automatiques. Si une valeur dépasse ce qui a été déclaré à l'onboarding, une erreur s'affiche immédiatement.

❌ Exemple : Restaurant déclaré pour 60 couverts maximum → impossibe de créer une offre avec capacity = 61.
Message : "Capacité dépassée. Votre établissement peut accueillir 60 personnes maximum selon votre profil."

5.1 Table des contraintes par champ

Champ onboarding (ActivityDetails) Champ offre Règle Types concernés
capacite_max / capacite_salle capacity offre ≤ onboarding Tous
max_participants max_group_size offre ≤ onboarding Activité, Éco-tour
niveaux (tableau) niveau_requis offre ⊂ onboarding Activité outdoor, Randonnée
langues_guides (tableau) langue_guide offre ⊂ onboarding Éco-tour, Culture
capacite_vehicule capacity offre ≤ onboarding Transport
nb_chambres / nb_tentes nb d'offres créées pour ce sous-type offres ≤ onboarding Hébergement

5.2 Où la validation se fait

🖥 Frontend (temps réel)

  • Erreur sous le champ dès la saisie
  • Valeur max affichée comme hint
  • Champ rouge si dépassement
  • Bouton "Suivant" désactivé si erreur

🔧 Backend (avant sauvegarde)

  • Charge l'activité liée (MongoDB)
  • Vérifie toutes les contraintes
  • Lève BadRequestException si dépassement
  • Garantit la cohérence même si le frontend est contourné

6. Calendrier de disponibilité

À l'étape 4, le prestataire définit quand son offre est disponible. 4 modes :

📅 Mode A — Période fixe
Pour une offre saisonnière
▸ Date de début ▸ Date de fin ▸ Jours bloqués (exceptions) Ex : ouvert du 1er juin au 30 sept
🔁 Mode B — Jours récurrents
Pour une activité hebdomadaire régulière
▸ Jours de la semaine (Lun–Dim) ▸ Créneau horaire (début → fin) ▸ Jours bloqués (congés, fêtes) Ex : Sam + Dim, 9h → 17h
📌 Mode C — Dates spécifiques
Pour des événements ponctuels
▸ Sélection multiple sur calendrier ▸ Créneau horaire par date (optionnel) Ex : 14 juil / 15 juil / 20 juil
♾ Mode D — Toujours disponible
Sans contrainte calendaire
▸ Aucune date à définir ▸ Le voyageur propose une date ▸ Le prestataire confirme Adapté aux offres sur demande
Type d'offreMode recommandé
Hébergement (nuit)Mode A (saison) ou D
Restaurant / Table d'hôteMode B (jours d'ouverture)
Activité outdoor / RandonnéeMode B (week-ends) ou C (dates précises)
Éco-tour multi-joursMode C (départs planifiés) ou A
Atelier artisanatMode C (sessions planifiées)
VolontariatMode A (campagne) ou D
Transport écoMode D (sur demande)

7. Mode de confirmation

✅

Confirmation instantanée

Réservation confirmée automatiquement dès soumission.

Backend : confirmation_mode = 'instant'

⏳

Sur demande

Le prestataire accepte ou refuse dans un délai défini (24h/48h/72h).

Backend : confirmation_mode = 'manual'
confirmation_deadline_hours

💰

Avec acompte

X% payé en ligne pour bloquer. Solde versé sur place ou à la date.

Backend : confirmation_mode = 'conditional'
deposit_percentage

✅ Ces 3 modes et leurs colonnes (confirmation_mode, confirmation_deadline_hours, deposit_percentage) existent déjà dans l'entité Offer.

9. Résumé — ce qu'il faut implémenter

🔧 Backend

  • +7 colonnes dans Offer entity
  • Créer entité Reservation
  • Service de validation des contraintes
  • Endpoints : créer offre, liste offres org, créer réservation, changer statut réservation
  • Notification prestataire à la réservation

🖥 Frontend

  • Formulaire d'offre 5 étapes
  • Sélecteur d'activité (étape 0)
  • Sélecteur multi-sous-types + mode variantes/package
  • Champs dynamiques par sous-type (couche 2)
  • Composant calendrier disponibilité
  • Erreurs de contrainte en temps réel
  • Page publique de l'offre
  • Formulaire de réservation voyageur
  • Section "Mes offres" dans le profil prestataire
✅ Ce qui est déjà prêt : L'entité Offer est quasi-complète (7 des 14 colonnes nécessaires existent). Les ProviderActivities contiennent les données pour le pré-remplissage et les contraintes. L'architecture Provider → Organisation → Activité est opérationnelle. Le gros du travail = formulaire multi-étapes frontend + validation contraintes + entité Reservation.
